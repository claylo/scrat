# `scrat init` Command Design

## Goal

Add a `scrat init` command that walks users through project discovery
and generates a config file (`scrat.toml` or `scrat.yaml`)
with their choices filled in.

## Architecture

Follows the thin-CLI / fat-core pattern.

**Core** (`scrat-core/src/init.rs`):

- `plan_init(project_root) -> InitPlan` — runs ecosystem detection,
  version strategy detection, release branch detection, tool probing.
  Returns everything the CLI needs to present as defaults.
- `generate_config(selections, style) -> String` — takes the user's
  confirmed/overridden values plus a style (documented or minimal)
  and format (TOML or YAML), returns the config file content as a string.
  Template-driven, not serde serialization — because the documented style
  needs comments, examples, and selective uncommenting.

**CLI** (`scrat/src/commands/init.rs`):

- Checks for existing config file.
  If found: prompt merge/overwrite/exit.
- Calls `plan_init()` to get detected defaults.
- Walks user through prompts with detected values as defaults.
- Calls `generate_config()` with selections.
- Writes the file.

## InitPlan (core returns this)

```rust
pub struct InitPlan {
    /// Detected ecosystem, if any.
    pub ecosystem: Option<Ecosystem>,
    /// Detected release branch (main, master, or None).
    pub release_branch: Option<String>,
    /// Detected version strategy.
    pub version_strategy: Option<VersionStrategy>,
    /// Detected changelog tool.
    pub changelog_tool: Option<ChangelogTool>,
    /// Detected tools (test cmd, build cmd, etc.).
    pub tools: Option<ToolDetection>,
    /// Whether an existing config file was found, and its path.
    pub existing_config: Option<String>,
}
```

No user interaction in core.
Core detects, CLI prompts.

## InitSelections (CLI builds this from prompts)

```rust
pub struct InitSelections {
    pub format: ConfigFormat,          // Toml or Yaml
    pub style: ConfigStyle,            // Documented or Minimal
    pub ecosystem: Option<Ecosystem>,  // None = omit (auto-detect is fine)
    pub release_branch: Option<String>,// None = omit (auto-detect is fine)
    pub github_release: bool,          // default true
    pub draft: bool,                   // default true
}

pub enum ConfigFormat { Toml, Yaml }
pub enum ConfigStyle { Documented, Minimal }
```

## Interactive Prompt Flow

~6 prompts, all with detected defaults pre-selected:

1. **Existing config** (only if found):
   "Found existing config at ./scrat.toml. What do you want to do?"
   → Merge/update | Overwrite | Exit

2. **Config format**:
   "Config format?" → TOML (recommended) | YAML

3. **Config style**:
   "Config style?" → Documented (recommended) | Minimal
   Documented = all fields present with comments and examples.
   Minimal = only non-default values, no comments.

4. **Ecosystem**:
   "Detected: Rust. Correct?" → Yes | Change to Node | Change to Generic
   (If nothing detected: same prompt as the existing ecosystem selection)

5. **GitHub releases**:
   "Create GitHub releases?" → Yes, as drafts (recommended) | Yes, published | No

6. **Confirm write**:
   "Write to ./scrat.toml?" → Yes | Change path | Cancel

If merging with existing config:
prompts 2-5 show existing values as defaults instead of detected values.

## Config Generation

Template-driven, not serde serialization.
The documented style needs:

- Comment blocks explaining each section
- User's active values uncommented
- Default/unused values shown as comments
- Hook examples with realistic commands
- YAML values quoted when they contain special characters
  (`{version}`, colons, brackets)

### Documented TOML Example

```toml
# scrat configuration
# Generated by `scrat init`
#
# Docs: https://github.com/claylo/scrat

# ─── Logging ─────────────────────────────────────────────────────────
# Values: debug, info, warn, error
# Override at runtime with --verbose (-v) or --quiet (-q)
# log_level = "info"

# ─── Project ─────────────────────────────────────────────────────────
# Auto-detected from Cargo.toml / package.json. Override if needed.
[project]
type = "rust"
# release_branch = "main"

# ─── Version Strategy ────────────────────────────────────────────────
# Auto-detected from cliff.toml / cog.toml presence.
# Values: conventional-commits, interactive, explicit
# [version]
# strategy = "conventional-commits"

# ─── Command Overrides ───────────────────────────────────────────────
# Defaults are set per ecosystem. Override individual commands here.
# [commands]
# test = "cargo nextest run"
# build = "cargo build --release"
# publish = "cargo publish"

# ─── Release ─────────────────────────────────────────────────────────
[release]
github_release = true
draft = true
# title = "{tag}"
# assets = ["release-card.png", "checksums.txt"]
# notes_template = "templates/release-notes.tera"
# discussion_category = "Announcements"

# ─── Hooks ────────────────────────────────────────────────────────────
# Shell commands that run at each phase of `scrat ship`.
# Variables: {version}, {prev_version}, {tag}, {changelog_path},
#            {owner}, {repo}
#
# Commands run in parallel by default.
#   sync:   prefix = barrier (wait for prior, run alone, then continue)
#   filter: prefix = sync + JSON stdin/stdout for pipeline mutation
#
# Phase order: pre_ship → test → bump → publish → git → release → post_ship
#
# [hooks]
# pre_ship = []
# post_ship = []
# pre_test = []
# post_test = []
# pre_bump = []
# post_bump = [
#     "generate-release-card --version {version} -o release-card.png",
# ]
# pre_publish = []
# post_publish = []
# pre_tag = []
# post_tag = []
# pre_release = []
# post_release = [
#     "sync: gh release edit {tag} --draft=false",
#     "notify-slack 'Released {owner}/{repo} {tag}'",
# ]

# ─── Ship Behavior ───────────────────────────────────────────────────
# [ship]
# confirm = true    # Prompt before executing. Use --yes/-y to skip.
```

## Merge Behavior

When merging with an existing config:

- Read existing config with Figment (same as normal config loading)
- Use existing values as prompt defaults instead of detected values
- Generate fresh config file with merged selections
- The old file is replaced entirely (not patched in-place) —
  this ensures the documented style is clean and complete

## Error Handling

- No git repo: warn but continue (some fields like release_branch
  won't be detected)
- No ecosystem detected: prompt selection (same as existing flow)
- Write failure: report error with path, suggest permissions fix
- User cancels: exit cleanly with "Init cancelled."

## Testing

- `plan_init()` unit tests with temp dirs containing various marker files
- `generate_config()` snapshot tests for all 4 combos
  (toml/yaml x documented/minimal)
- Integration test: `scrat init --yes` with pre-set answers produces
  expected config file

## CLI Args

```rust
#[derive(Args, Debug, Default)]
pub struct InitArgs {
    /// Config format to generate
    #[arg(long, value_enum)]
    pub format: Option<ConfigFormat>,

    /// Config style: documented (all options with comments) or minimal
    #[arg(long, value_enum)]
    pub style: Option<ConfigStyle>,

    /// Write config without prompting (uses detected defaults)
    #[arg(long, short = 'y')]
    pub yes: bool,

    /// Output path (default: ./scrat.toml or ./scrat.yaml)
    #[arg(long, short = 'o')]
    pub output: Option<String>,
}
```

`--yes` skips all prompts, uses detected defaults, writes documented TOML.
Useful for CI or scripting: `scrat init --yes --format yaml --style minimal`.

## Scope Boundaries

**In scope:**

- Interactive prompt flow with detected defaults
- Config file generation (TOML + YAML, documented + minimal)
- Existing config handling (merge/overwrite/exit)
- `--yes` for non-interactive use

**Out of scope (future):**

- `scrat config --show` to display effective merged config
- `scrat config --validate` to check config against schema
- `inherit: [path]` directive (separate feature)
- Config profiles (XDG central → per-hat → project-local)
